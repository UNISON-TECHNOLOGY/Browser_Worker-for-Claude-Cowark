# コンテンツ設計スキル（C×P×T）

記事は **内容（C: 何を書くか）× 構成パターン（P: どう組むか）× 加工方法（T: 情報をどう料理するか）** の3軸で設計する。演出（画像・対話・図解）はさらに独立したレイヤー。

## 大原則

- **リードに全結論を入れる**。読者は冒頭で読む価値を判断する。結論を隠して引っ張る構成は使わない
- 1記事1メッセージ。詰め込むくらいなら分割する

## 構成パターン（P）

| ID | パターン | 骨格 | 向き |
|----|---------|------|------|
| P1 | PREP | 結論→理由→具体例→結論 | 主張系の汎用 |
| P2 | ステップ | 結論→手順1..n→まとめ（3〜7步、各步に図解） | ハウツー |
| P3 | 並列比較 | 結論→比較表先出し→軸ごと詳細→向き不向き | 選択支援 |
| P4 | 逆三角形 | 最重要ファクト→背景→詳細→見解 | 速報。途中離脱でも成立 |
| P5 | リスト | 結論+総数→項目1..n（形式統一）→比較表→CTA | ◯選。数字タイトルでCTR向上 |
| P6 | Before/After | 結論→問題→施策→変化(数値)→学び | 体験記 |
| P7 | Q&A | 結論→Q1A1..→まとめ | 検索意図直結・FAQスキーマ対応 |
| P8 | 対話劇 | 結論→掛け合いで展開→オチ→ポイント整理 | キャラ運用時の主力 |
| P9 | ケーススタディ | 結論→状況→施策→結果(数値)→分析→教訓 | 事例 |
| P10 | Problem/Solution | 結論→問題定義→原因→解決策→実行例 | 課題駆動 |

## 加工方法（T）— 同じネタでも料理で変わる

| ID | 加工 | 一言定義 |
|----|------|---------|
| T1 | 整理 | 散在する事実を構造化 |
| T2 | 翻訳 | 専門情報を読者の言葉に変換 |
| T3 | 解釈 | 事実に「だからこう読める」を加える |
| T4 | 検証 | 「本当？」を実際に確かめる |
| T5 | 実践 | やってみた結果を見せる |
| T6 | 比較 | 同一基準で並べて違いを出す |
| T7 | 対比 | 常識と事実のギャップを際立たせる |

- **ローテーション**: 直近の記事と同じ T が2回続いたら、3回目は使用頻度の低い T を選ぶ（マンネリ防止）

## 品質ガード

**誇張ドリフト注意（特に P6）**: 変化を過剰演出（「人生が変わった」「別物になった」）、原因曖昧なまま結果だけ強調、読者に過度な期待を持たせる — に該当したら P4 へ変更を検討。トーンは「淡々と事実を見せる」が基本。

## 演出レイヤー（構成とは独立）

- **本文（情報パート）は落ち着いたトーンで書く**。演出は導入とブレイクの味付け
- 実画面スクショは1記事3〜8枚・キャプション必須（「何の画面で何を見るか」を1文）
- 図解は「抽象概念・比較・まとめ」の可視化のみ。装飾に使わない
- 外部リンクは公式ソースのみを記事末尾「参考」に集約。価格・スペックには「執筆時点（YYYY年M月）」を付す

## ネタ帳・ストック運用（量産の型）

- ネタは `knowledge/content-queue.md` に **Inbox →（検証済み）Ready →（使用後）Used** で管理
- ストック目標を決めて充足する（例: 記事4本分・SNS 3日分）。目標割れが「作るべきサイン」
- 公開後はメトリクスを記録し、次のテーマ選定に反映する（伸びた C×P×T の組み合わせを増やす）
