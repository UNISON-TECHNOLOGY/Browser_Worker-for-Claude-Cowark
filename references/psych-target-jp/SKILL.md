# ターゲット心理プロファイリング・スキル（日本データベース）

## 生成前の3軸判定（この3つを決めてから書く）

1. **不安の核は何か** → 先回り解消文を1行用意する
2. **意思決定スタイル**（慎重型/直感型） → 慎重型=数値・出典・検証可能な事実、直感型=冒頭のベネフィット一撃。両対応は「冒頭は感情、本文は数値」の二層構造
3. **関係段階**（初回/継続/面談後） → 敬語・人称の距離を設定

ペルソナはデモグラだけで作らない — 「意思決定スタイル」「不安の核」「規範（べき思考の中身）」の3心理変数を必ず持たせる（同年齢でも心理は正反対）。

## 読み手タイプ別判断表（日本の実証データつき）

| 読み手 | データ | 書き方 |
|---|---|---|
| スカウト受信者（全般） | 好印象=経歴の具体箇所への言及、悪印象=差し替えテンプレ（エン・ジャパン3,800人調査） | **冒頭2文以内に「その人固有の事実→なぜ声をかけたか」**。読んでいないのに「拝見しました」は最大の不信要因 |
| 未経験・不安層 | 「研修・受け入れ体制」明記で心理ハードル低下 | 要件より先に受け入れ体制・失敗許容を書く。要件リスト先頭は破局視を誘発 |
| ハイクラス | 逆 | 裁量・期待・課題の大きさを先に |
| Z世代・20代 | 「安定」49.9%で6年連続1位。「ネタバレ就活」（事前に全部知りたい）。勤務地確約希望88.8% | **予見可能性を明示**（1日の流れ・入社後3ヶ月・選考プロセス全体像・勤務地断定）。「圧倒的成長」「修羅場」系は離脱要因 |
| ITエンジニア | 最大不安=スキル陳腐化46.5%（パーソル総研）。開封率も低い（〜50%） | 年収より先に技術スタックの新しさ・学習投資・技術選定への関与。「安定した保守案件」推しは不安の核心を刺激。件名の固有性が最優先 |
| 転職検討層（一般） | 不満でなく「現職では解消されない」という予期が転職を起動（パーソル×中原1.2万人調査） | 現職否定ではなく「現職では構造的に得にくいものが、当社では得られる理由」を書く |

## CBT応用 — 読み手のブレーキ（認知の歪み）を外す書き方

読み手を診断するのではなく、**応募・購買を妨げる典型的な思考パターンに文面で先回りする**:

| 読み手のブレーキ | 文面での外し方 |
|---|---|
| 全か無か（「要件を全部満たさないと応募できない」） | 必須要件を最小化し「**いずれかに当てはまれば**」と書く |
| 破局視（「転職に失敗したら終わり」） | 小さく試せる次の一歩を提示（「カジュアル面談だけ」「情報収集の返信も歓迎」「現職のまま話だけ」） |
| べき思考（「3年は続けるべき」） | 規範を否定せず選択肢を並置（「続ける選択も、環境を変える選択も」） |
| 不安全般 | **煽らず先回り解消**: CTA直前に想定不安への回答1行（「返信=応募ではありません」）。恐怖訴求（「今動かないと市場価値が下がる」）は禁止 |

## 日本語の人称・敬語・断定の判断表

| 要素 | 基準 |
|---|---|
| 二人称 | スカウトでは「あなた」を避け「◯◯様」+動作主省略（「〜のご経験を拝見し」）で自分ごと化を得る。「あなた」連呼は目上・ハイクラスに失礼リスク。LP・広告は「あなた」より状況描写型で自己投影を誘う（「自分の技術、古びていないか」） |
| 断定/緩和 | **事実は断定、心理は緩和** — 会社が保証できる事実（給与・制度・数値）は「〜です」と断定。相手の内面は必ずヘッジ（「もし〜でしたら」）。逆転（事実を曖昧に・相手の心理を断定「転職をお考えのはずです」）が最も不信を招く |
| 敬語距離 | 初回=完全な敬体・二重敬語なし。2通目以降・面談後=話し言葉を一部混ぜて一段ずつ距離を縮める。全接点で慇懃な定型敬語を維持し続けるのも関係構築を妨げる |
| 検証可能性 | 形容詞だけの訴求（「アットホーム」「働きやすい」）は慎重型に不信。定量に置換（「月平均残業12時間」）。求人ABテストで応募3〜4倍の差が出るのはタイトル×冒頭3〜4行 |

## 倫理境界（この線を越えない）

**実務テスト: その心理効果は「事実を分かりやすく伝えた結果」か、「事実を歪めて/隠して作った感情」か。前者のみ正当。**

- CBT知見は読み手の不安の**解消**に使う。不安の意図的な**増幅**（恐怖マーケ）は越境で禁止
- 偽の希少性・根拠なき優遇演出・「あと3枠」等の虚構は景表法（優良誤認・有利誤認）+消費者庁ダークパターン規制の対象
- 読み手を「診断」する物言い（「あなたは◯◯タイプなので」）をしない — プロファイルは書き手側の内部判断に留める
- 送信前は pre-send-verifier の監査対象（この基準への適合も監査観点になる）
