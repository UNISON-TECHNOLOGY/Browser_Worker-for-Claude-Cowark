# CRO/ABテスト スキル
**対象領域: 採用LP・応募フォーム・スカウト/配信文面（日本語圏）**

## 改善の優先順位（インパクト順）

1. **ファーストビュー** — 3秒で「誰向けか・何が得られるか・次に何をするか」が伝わるか。CTAは1画面に1つ
2. **CTA（ボタン）** — 動詞で始める具体文言（「応募する」より「30秒で応募を完了する」）、色よりも文言と位置
3. **フォーム** — 項目数が最大の敵。1項目削減の効果は装飾全部より大きい。必須は最小限・エラーは項目の横に即時表示・スマホでの入力しやすさ最優先
4. **社会的証明** — 実名の声・数字（応募者数・定着率）・第三者評価。作り物っぽい証言は逆効果
5. **不安の除去** — 「応募したら何が起きるか」（連絡方法・所要時間・断ってもよいこと）を CTA 近くに明記

## 仮説の立て方（型）

```
現状: <数値: 例 応募率1.2%> の <ボトルネック: 例 フォーム到達後の離脱67%>
仮説: <変更> すれば <対象ユーザー> の <障壁> が減り <指標> が上がる
検証: <期間・母数・判定基準を事前に固定>
```

- 仮説なしの「なんとなく変更」を提案しない。1回のテストで変えるのは1要素
- 母数が小さい（週の応募が2桁未満等）場合は AB テストでなく**逐次改善+前後比較**を提案する（統計的に差が出ない期間テストは時間の無駄）
- 判定基準・期間は**開始前に固定**し、途中で動かさない（覗き見による誤判定の防止）

## スカウト・配信文面のABテスト

- 変える要素は1つ: 件名（copywriting・13字）/ 冒頭1文 / CTA / 送信時間帯
- 送信数を半分に割り、同条件のターゲットで比較。結果は knowledge/analytics/ に記録し outcome-verifier で集計
- 勝ちパターンはテンプレの正本（knowledge のスカウトプレイブック等）に反映して初めて完了（メモだけ残さない — I-5 と同じ思想）

## 禁止・注意

- ダークパターン禁止 — 偽のカウントダウン・在庫僅少の偽装・解約導線隠しは提案しない（信頼毀損 + 景表法リスク。ad-compliance-jp 参照）
- 改善案のモックアップ化は page-improve 部品（design-artisan → design-critic）、実ページへの反映はタスク開始手順のゲート下で
- 数値の解釈で相関と因果を混同しない（「変えた週に増えた」だけなら要因未特定と明記）
