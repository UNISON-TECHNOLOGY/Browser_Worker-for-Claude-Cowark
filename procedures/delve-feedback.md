ユーザーのフィードバック（$ARGUMENTS または直前の会話での評価・修正指示）を学習記録に変換してください。

## 手順

1. `knowledge/feedback/lessons.md` を Read（なければ下の雛形で新規作成）
2. フィードバックを**再利用可能な規則**に変換して追記する。単発の事実ではなく「次も適用できる形」にすること:
   - ❌「このボタンの色を変えた」
   - ⭕「NG: 青系CTAは避ける（ブランドのオレンジと競合するため）— 2026-07-22, LPモックアップ」
3. 既存エントリと矛盾する場合は古い方に取り消し線を引き、新しい規則を正とする（黙って上書きしない）
4. 記録後、「学習しました: <規則の要約>」と1行で報告する

## lessons.md の雛形

```markdown
# 成果物フィードバック学習記録

生成系エージェント（deliverable-writer / design-artisan / design-critic）は作業前にこのファイルを必ず読むこと。
NG は禁止事項として、OK は優先パターンとして扱う。矛盾時は日付の新しい方が正。

## デザイン（レイアウト・色・モーション）
<!-- 例: NG: ファーストビューに要素4つ以上（ごちゃつくとの評価）— 2026-07-22 -->

## 文章・トーン
<!-- 例: OK: 数値は「◯◯万円」表記（カンマ区切りよりこちらを好む）— 2026-07-22 -->

## レポート形式
<!-- 例: NG: サマリーが5行超（3行で言い切ってほしい）— 2026-07-22 -->

## その他
```

## ガード連動（禁止・許可のフィードバック）

フィードバックが成果物の好みではなく**ガードの発火条件**に関するものなら、lessons.md ではなく該当の設定ファイルに反映する（hook は毎回ファイルを読み直すため、追記した次の操作から即効く）:

| ユーザーの発言例 | 反映先 | AIが直接書いてよいか |
|---|---|---|
| 「このサイト/ページは触るな・見るな」 | `knowledge/config/url-denylist.txt` にパターン追記 | ✅ 即追記（強める方向） |
| 「この操作・文言も危険扱いにして」 | `knowledge/config/money-watchlist.txt` に追記 | ✅ 即追記（強める方向） |
| 「この画面の検知は誤爆。次から出すな」 | `knowledge/config/money-suppress.txt` に追記 | ✅ 追記可。ただし抑制は緩める方向なので、追記したパターンを復唱してユーザーに確認する |
| 「このサイトの課金操作を許可する」 | `knowledge/config/url-allowlist.txt` | ❌ **AIは書かない**（金銭開放は人間の手編集のみ。編集手順を案内する） |

- 追記する行には必ず由来コメントを添える: `# 2026-07-23 ユーザー指示「<発言の要約>」` の直後にパターン行
- ガード連動で反映した場合も lessons.md に1行の記録（「ガード設定に反映: <内容>」）を残し、二重管理はしない（正はconfig側）

## 分類ルール

- 対象カテゴリ（デザイン/文章/レポート形式/その他）に振り分ける
- 各エントリは1行: `OK|NG: <規則> （<理由>）— <日付>, <対象成果物>`
- 好みか普遍原則か迷ったら好みとして記録（普遍原則はスキル側に既にある）
- 1カテゴリ20行を超えたら、古い・重複エントリの整理を提案する
