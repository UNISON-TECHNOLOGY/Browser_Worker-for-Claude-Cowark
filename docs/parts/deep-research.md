徹底モードで実行してください: $ARGUMENTS

## 原則

- このコマンドはユーザーが明示的に深さを求めたときだけ発火する（trivial なタスクに使わない）
- **Workflow ツールが使える環境では Workflow で組む**（ループ・並列・検証を決定論的に制御）。無ければ Agent ツールの逐次・並列呼び出しで代替
- 開始前に概算規模（エージェント数の目安）を1行でユーザーに伝える

## model / effort の配分規則（Workflow の agent() 呼び出し・Agent ツールとも共通）

| 役割 | model | effort | 該当例 |
|---|---|---|---|
| ワーカー（発見・執筆・変換） | sonnet | medium | loop-until-dry の発見役、ペルソナレビュアー、原稿生成 |
| 検証・審査・裁定 | opus | medium | 敵対的検証の反証役、トーナメント審査、最終裁定 |
| 設計の壁打ち | strategy-advisor（定義済み） | - | 実行前の計画セカンドオピニオン |
| ビジュアル生成 | fable（不可なら sonnet） | high | モックアップHTML生成、デザイン案（定義済み design-artisan を優先使用） |
| 機械的な軽作業（分類・場合分け・整形） | haiku（不安があれば sonnet） | low | 分類・振り分け、リスト整形、突合、重複排除 |

- 事前定義エージェント（deliverable-writer / design-artisan / design-critic / strategy-advisor / pre-send-verifier）は定義ファイル側に model/effort を持つため、呼び出し時の上書きは不要
- 動的生成エージェント（ペルソナ等）はこの表に従って `agent(prompt, {model, effort})` を指定する

## パターン集（タスクに合わせて選択・合成）

### 1. 動的ペルソナレビュー（専門エージェントの動的生成）
対象の実データからレビュアーを**その場で設計**する:
- LP改善なら: ターゲット読者ペルソナ（knowledge のトークン・コンテンツから具体化。例「27歳・事務職・転職に不安」）/ 競合マーケター視点 / 初見3秒視点 — の3〜5人を独立並列でレビュー
- 各ペルソナは事前定義エージェントではなく、タスク文脈から書いた専門プロンプトで Agent を起動する
- 相反する指摘は design-critic（事前定義）が裁定する

### 2. loop-until-dry（発見系: 課題洗い出し・競合調査・リンク検査）
- 発見エージェントを並列で回し、新規発見が2ラウンド連続ゼロになるまで継続
- 発見物は毎ラウンド既知リストと突合して重複排除（却下済みも再浮上させない）

### 3. 敵対的検証（確度が要る主張・診断）
- 重要な発見・改善提案ごとに「反証を試みる」検証エージェントを独立に3体、多数決で採否
- 検証を生き残ったものだけをレポートに載せ、落ちたものは「不確実」として脚注へ

### 4. 生成トーナメント（品質最優先の成果物）
- 異なる戦略（例: 実績訴求先行 / 共感先行 / オファー先行）で独立に3案生成 → 動的ペルソナ審査で採点 → 勝者をベースに敗者の良い部分を移植して最終案

## 組み合わせの例

「このLPを徹底改善」= 課題洗い出し(loop-until-dry) → 敵対的検証で課題を確定 → 生成トーナメントでモックアップ3案 → 動的ペルソナレビュー → design-critic 裁定 → 通常の目視検証へ合流

## 締め

- 結果は通常のレポート規範（テンプレート・アーティファクト発行）に合流させる
- 「何体のエージェントで・何を捨てて・何が生き残ったか」のプロセスサマリーを必ず添える（過程の透明性）
- knowledge/logs/ に実行記録を残す
