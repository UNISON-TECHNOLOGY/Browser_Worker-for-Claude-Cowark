# SNS共通運用フロー（正本）

各媒体パック（procedures/delve-sns-*.md）がこのフローを Read して従う。媒体固有の見える範囲・制約はパック側に記載。

**予約・投稿はブラウザ変更操作なのでタスク開始手順（procedures/delve-start.md）のゲート手続きが必須。**

## フェーズ構成（3段階、各段でユーザー確認）

### ① 実態照合 — 「記録を信用しない。ライブが唯一の真実」

1. プラットフォーム側の**実際の予約キュー・投稿済みタイムライン**をブラウザで確認する（read_page）
2. ワークスペースの記録（`knowledge/sns/<platform>/queue.md` や DB）と突き合わせ、ズレを実態に合わせて補正する
   - ⚠️ 記録上の空き枠だけで設計すると**二重予約事故**が起きる（原典リポジトリで実際に発生した教訓）
   - 投稿忘れ（予定はあるが実TLに無い）も検知し、即時投稿するか破棄するか確認
3. メトリクス収集: 投稿後3日以内かつ前回取得から24h超のものだけ計測（**3日超は計測しない** — 数字が動かないため）。knowledge/data/delvework.db の sns_posts / sns_metrics に記録（DBが無ければ templates/db-schema.sql で初期化。python 標準の sqlite3 を使用）

### ② 改善分析 — 数字を次の生産に戻す

1. 直近14日のメトリクスから読み取る:
   - **反応ゼロ投稿**の共通点（同じ内容種別×構成の組み合わせを次で避ける）
   - **内容種別の偏り**（偏っていたら未使用の種別を1枠で試す）
   - **勝ちパターン**（最強の組み合わせを再現、最弱を回避）
2. ストック目標との差分から今回の作成本数を確定（目標例: X=3日分12枠 / note=下書き4本）
3. 作成計画（本数・枠・テーマ・種別）を提示 → **ユーザー確認**

### ③ ストック充足 — 生成→承認→予約

1. ネタは `knowledge/sns/<platform>/queue.md`（Inbox→Ready→Used）の Ready から優先消化。不足時はリサーチから補充
2. 生成は references/ の該当ファイルを Read して適用: 構成は content-design（C×P×T）、日本の投稿慣習は sns-jp、社員・現場エピソード系は storytelling、コピー・件名は copywriting、トーンは `knowledge/feedback/lessons.md`（ワークスペース蓄積。なければスキップ）、法規は ad-compliance-jp（**公開前チェック必須**）
3. **1投稿ごとにユーザー承認 → ブラウザで予約投稿**（Delvework ゲート下: /タスク開始 済み・変更前記録済みであること）
4. 予約後に記録を更新（queue の Ready→Used、予約一覧への追記）
5. 全枠充足まで 1→4 を繰り返し、最後に「充足状況+今回の学び」を session-log に記録

## 初回実行時（ナレッジがない場合）

- `knowledge/sns/<platform>/` を作成し、queue.md（Inbox/Ready/Used の3セクション）と strategy.md（投稿枠・時間帯・トーン方針をユーザーにヒアリングして記録）を初期化する
- ストック目標をユーザーと合意して strategy.md に記録

## 注意

- 投稿文面の実績数値・引用は捏造禁止。時事ネタは炎上リスク審査（sns-jp の公開前チェック）
- プラットフォームの予約UIの構造は knowledge/sites/<platform>/ に Delvework 方式でマッピングして再利用する

## 実績記録の検証

定常運用の締め（実績記録）では outcome-verifier に投稿ログと実測数値を渡して集計・検証させ、結果を knowledge/analytics/ に記録してからダッシュボード更新につなぐ。
