# 無人運用ガイド（PC起動 → 自動で定常タスク）

「人間が席にいなくても定常タスクが回る」ための構成。**パスワードをAIや設定ファイルに保存しない**まま無人化するのが本ガイドの方針。

## 認証の3層設計

| 層 | 仕組み | 効果 |
|---|---|---|
| 1. セッション永続 | Claude in Chrome は実ブラウザのプロファイルで動く。一度ログインすれば cookie が再起動後も持続 | 通常運転ではログイン作業ゼロ・保管物ゼロ |
| 2. ブラウザ autofill | パスワードは Chrome/1Password 等のパスワードマネージャにのみ保存。セッション切れ時は AI がログインページを開き、**ブラウザが自動入力 → AI はログインボタンをクリックするだけ** | パスワードは AI を一度も通過しない（Credential Guard と両立） |
| 3. 2FA は人間へ通知 | ワンタイムコード・認証コードに遭遇したらタスクを中断し、ユーザーに通知。入力後に再開 | 2FA の安全性を壊さない |

### やってはいけないこと

- パスワードを registry.yaml / .env / スクリプト / メモに書く（平文保管は全面禁止）
- AI にパスワードをチャットで教える（会話ログに残る）
- 2FA を無効化して回避する

## セットアップ手順

1. **各媒体に一度手動でログイン**（「ログイン状態を保持」を必ずチェック）し、パスワードはブラウザのパスワードマネージャに保存
2. **Claude デスクトップをスタートアップ登録**: `Win+R` → `shell:startup` → Claude のショートカットを配置（PC起動で Cowork が立ち上がる）
3. **定常タスクを Cowork のスケジュール機能に登録**。実体は trigger 系 MCP ツール（`create_trigger` / `list_triggers` / `update_trigger` / `delete_trigger` / `send_later`。※`create_scheduled_task` という名前のツールは存在しない）。登録時の必須知識:
   - **cron は UTC 評価**。JST の希望時刻から9時間引く（例: 月曜 JST 9:00 → `0 0 * * 1`）。変換を誤ると9時間ズレて実行される
   - **最小間隔は1時間**。毎時系は登録時刻にアンカーされ、発火時刻には数分のジッターが乗る
   - **発火は新規セッションとして起動する**（登録した会話には戻らない）。元の会話に戻したい一回限りの用途は `send_later`
   - **発火プロンプトは短く**（例: 「<タスク名>やって。終わったらダッシュボード更新して」）。詳細手順は `tasks/<タスク名>.yaml` に置き、発火時に読ませる。trigger の job_config にはプロンプト全文が埋め込まれ肥大化するため
   - 登録例:
     - 毎週月曜 JST 9:00（`0 0 * * 1`） `/競合ウォッチ <競合URL...>`（競合定点観測）
     - 毎日 JST 8:00（`0 23 * * *`） `/SNS運用 x`（Xストック確認・充足）
     - 毎月1日 JST 9:00（`0 0 1 * *`） `/サイト診断 <自社URL>`（サイト健康診断）
4. **ブラウザの一意化**: 複数の Chrome（複数PC・複数プロファイル）が接続されていると、操作前のブラウザ選択で人間の応答待ちになり無人実行が停止する。無人運用する媒体の Chrome は1台に絞るか、`knowledge/config/browser.md` に優先ブラウザ（デバイス名/特徴）を記録しておき、タスク側はそれを読んで選択する
5. **通知経路の確認**: タスクがログイン切れ・2FA・承認待ちで停止したとき気づけるよう、デスクトップ/モバイルの通知を有効化

## タスク側の約束（全 delve コマンド共通）

- 巡回先で**ログアウト状態を検知したら**: 失敗として黙って終わらず、(a) ログインページを開いて autofill を試み（クリックのみ）、(b) それでも入れなければ「◯◯がログイン切れ。再ログインしてください」と通知して該当媒体をスキップ、残りを続行する
- **2FA・認証コード入力に遭遇したら**: 即中断してユーザー通知（Credential Guard もこれをブロックする）
- 無人実行でも**変更操作の承認設計は変えない**: 読み取り系（watch/media status/audit）は完全無人可。送信系（スカウト・SNS予約投稿）は「下書き・計画まで無人で準備し、承認待ちで停止」を基本とし、完全自動送信はユーザーが明示的にポリシーで許可した範囲のみ

## Slack 連携（承認と報告の非同期化）

ユーザーの claude.ai アカウントで Slack コネクタを接続している場合、定常タスクは Slack を通知・承認チャネルとして使う。

### 通知（そのまま送る）

- アラート（媒体残数・契約期限・競合の重要変更・ログイン切れ・2FA待ち）
- 定常タスクの完了サマリー（1メッセージに集約。タスクごとに分けて連投しない）
- ダッシュボード更新時のアーティファクトURL

### 非同期承認キュー（送信系の承認をSlackで）

エージェントは実行中に返信を待てないため、承認は実行を跨いで非同期に回す:

1. **提案フェーズ**: 定常タスクが送信候補（スカウト文面・SNS予約投稿案）を作成したら、
   `knowledge/approvals/pending.md` に承認ID付きで記録し、同内容を Slack に投稿して終了
   （投稿形式: 承認ID / 種別 / 宛先・枠 / 本文プレビュー / 「承認は ✅ リアクション or 『A-12 OK』と返信」）
2. **承認**: ユーザーが Slack で ✅ リアクション or 返信（修正指示の返信も可）
3. **実行フェーズ**: 次回の定常実行の冒頭で Slack の該当スレッドを読み、
   - 承認済み → ゲート手続きの上で送信し、pending.md を done に移動、Slack に完了報告
   - 修正指示 → 修正して再提案
   - 48時間反応なし → リマインドを1回だけ送る（連投しない）

### 原則

- **Slack承認で送信できるのは、事前にユーザーがポリシーで許可した種別のみ**（初回に「Slack承認で送信してよい操作」を合意して knowledge/approvals/policy.md に記録）
- 承認の正本は Slack ではなく pending.md（Slack は入力インターフェース）。二重送信防止のため送信前に必ず pending.md の状態を確認する
- Slack コネクタが無い環境では、従来どおりチャット内での承認にフォールバックする

## リスクと割り切り

- 実ブラウザのセッションを使う = **PC自体のセキュリティが土台**（OSログインパスワード・ディスク暗号化・離席ロックを前提とする）
- 共有PCでの無人運用は不可（セッション=本人性のため）
- セッション乗っ取りを疑う挙動（身に覚えのない通知等）があれば、全媒体のパスワード変更とセッション無効化を行う
